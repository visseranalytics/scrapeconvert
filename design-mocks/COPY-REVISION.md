# ScrapeConvert: Home Page Copy (Revision)

## What a fresh visitor should grasp in 5 seconds

ScrapeConvert is a free tool that pulls images off any website (one page, several pages, or a whole site), flags the duplicate copies, and converts the ones you pick to WebP, AVIF, PNG, or JPEG. The conversion runs in your own browser, so your image files are never uploaded or stored. No account, no ads. If you build or refresh websites, this is the tool that does the grabbing and the converting in one place instead of two.

Voice pass applied: no em dashes, no buzzwords, no three-part rhythm lines, no fragments posing as sentences, no formulaic openers. Plain outcomes lead. Technical terms (CORS, LCP, Core Web Vitals) sit in parentheses or in the FAQ. Honesty checks applied: conversion is stated as local with no server copies; scraping is never claimed to be local (the proxy is disclosed); no competitor is named, only "most tools."

## Product story 1: The scraper (lead with this)

This is the differentiator, so lead the body with it. Make the reach (one page, many pages, whole-site crawl) and the hard-case coverage (img tags and CSS backgrounds) explicit, because "reads the markup" made one reviewer assume img-only.

**Eyebrow:** Scraper
**Heading:** Find every image on a site, not just the easy ones.

**Intro:**
Give ScrapeConvert one URL, a list of URLs, or a sitemap. It pulls the images into a single grid, flags the duplicates, and hands your selection straight to the converter. No re-uploading and no copy-paste.

**Feature: One page, many pages, or the whole site**
Scrape a single page when that is all you need. Paste a list of URLs to pull from several pages at once. Or point it at a sitemap and it crawls the whole site, pulling images from every page it reaches. Most image tools can only do a single page, if they scrape at all.

**Feature: Finds images other tools miss**
It catches every visible `<img>` on the page, plus the images set through CSS `background-image`, which a lot of grabbers skip entirely. The images hiding in your stylesheets show up in the grid with the rest.

**Feature: Flags duplicates by content, not just filename**
The same photo often appears at several URLs and sizes, like a logo or a retina `@2x` version. ScrapeConvert flags those repeats so you download each image once instead of three times.

**Feature: Pick a batch and send it straight to the converter**
Filter by format or size, sort the grid, tick the images you want, and pass the whole selection into the converter in one move. Other grabbers stop at the download and leave the converting to you.

**Reassurance card (benefit-level, no proxy or CORS detail):**
ScrapeConvert pulls public images only and skips paid stock-photo sites, so the grab stays fair to the sites you point it at.

*Intent: every gap the reviewers flagged is now an explicit claim: multi-page, sitemap crawl, CSS backgrounds, content-based dedupe, batch handoff, and the "most tools only do a slice" contrast. CORS, proxy, and bot-check detail stay out and live in the FAQ.*

## Why optimized images (cut the 101, prove the payoff)

Reviewers already know what WebP is and found this section condescending. Shrink it, translate the jargon, and lead with the outcome a PM scans for.

**Eyebrow:** Why bother
**Heading:** Lighter images make for faster pages and better ranking.

**Body:**
Images are usually the heaviest part of a web page. A WebP file is typically 25-35% smaller than the same JPEG and 60-80% smaller than the same PNG, with no visible drop in quality. Lighter images load faster, search engines reward faster pages, and you pay less to serve them. For most sites this is the quickest speed win there is.

**Link:** Read the guide: WebP vs PNG vs JPEG

**Three supporting cards (jargon moved into parentheses):**

- **Faster pages**
  Fewer bytes per image means the page paints sooner, even on a phone. Your main image often decides how fast the page feels (its Largest Contentful Paint).

- **Better Google ranking**
  Search engines factor page speed into ranking (Core Web Vitals). Lighter images are one of the simplest ways to move that number.

- **Lower hosting bills**
  Smaller files mean less data served on every visit, which trims your hosting and CDN costs over time.

**Demo caption (if the simulator stays):**
This is an illustration of how a lighter file finishes first, not a live benchmark.

*Intent: keeps real numbers (the ranges the brief allows), translates CWV, LCP, and CDN into plain outcomes with the term in parentheses, and stops the section from teaching developers what they already know. The caption keeps the simulator honest, since a developer opening devtools would otherwise read a fabricated demo as a credibility own-goal.*

## FAQ

This is where the honest nuance lives: the proxy, the limits on "any site," dedupe mechanics, formats, scale, and the data flow.

**Is it free?**
Yes, and it is open source under the MIT license. There are no file limits and no daily caps, and nothing is gated. If it saves you time, there is a buy-me-a-coffee link, but it is optional.

**Do you store my images?**
No. Conversion runs in your browser, so those files never reach us. Scraping is different: the page and image fetches pass through a proxy on our side so your browser can reach other sites, but we do not keep them. Nothing is written to a database or a storage bucket.

**Can it really pull from any site?**
It works on most public sites. To stay respectful and safe, the scraper skips paid stock-photo sites, refuses private and internal addresses, and rate-limits requests. A small bot check runs once per session so it cannot be turned into an abuse tool. Heavily protected sites may block automated fetches.

**How big a site can it handle?**
It is built for real sites with many pages and thousands of images, including sitemaps with hundreds of URLs. Very large crawls run slower because of the rate limits above, but they will not fall over.

**Which formats does it convert to?**
WebP, AVIF, PNG, and JPEG, all available today. WebP is usually the smallest common format for the web, and AVIF is often smaller still.

**How does duplicate detection work?**
It compares the image content, not just the filename, so it catches the same picture served at different URLs and sizes, including retina `@2x` and responsive variants.

**Why does scraping need a proxy?**
Browsers enforce CORS, which stops a page from fetching content on other domains. A small proxy sits between your browser and the target site so the request can go through. That is the only thing it does, and it only touches scraping fetches, never your conversions.

**Is the scraping respectful?**
We try hard to make it so. It pulls public images only, skips paid stock sites, refuses private addresses, and rate-limits requests so it stays light on the sites you point it at.

**Can I self-host it?**
Yes. The app and the proxy both deploy on Cloudflare. Clone the repo, set up your own Worker, and you have a private instance with your own rate limits. The README walks through it.

**Is there a browser extension?**
Yes, there is a free Chrome extension that scrapes the tab you are already on, so you do not have to paste a URL for pages you are browsing.

*Intent: reconciles the hero's "any site" with reality ("most public sites," with the limits stated), answers the developer questions about scale, formats, AVIF, and dedupe mechanics, and keeps all the proxy, CORS, Cloudflare, and bot-check detail here where it belongs. AVIF is handled honestly rather than ignored.*
